## معرفی

فیلم‌باز یک پلتفرم ماژولار مبتنی بر Django برای مرور و مدیریت فیلم و سریال است. این پروژه فراتر از یک کاتالوگ ساده طراحی شده؛ شامل سیستم کامل مدیریت کاربران، موتور جستجوی فازی مبتنی بر Trigram های PostgreSQL، لایه کشینگ Redis، و چت پشتیبانی بلادرنگ ساخته‌شده با Django Channels و WebSocket است — همه در قالب Docker کانتینرایز شده و برای استقرار Production پیکربندی شده‌اند.

---

## تکنولوژی‌های استفاده‌شده

| لایه | تکنولوژی |
|---|---|
| بک‌اند | Django + Django REST Framework |
| سرور HTTP | Gunicorn (WSGI) |
| سرور WebSocket | Daphne (ASGI) |
| ارتباط بلادرنگ | Django Channels |
| دیتابیس | PostgreSQL |
| کش و Channel Layer | Redis |
| صف وظایف | Celery Worker + Celery Beat |
| پیام‌رسان | RabbitMQ |
| پروکسی معکوس | Nginx |
| کانتینرایزیشن | Docker + Docker Compose |

## معماری پروژه

### ترکیب WSGI + ASGI

فیلم‌باز از دو سرور موازی برای مدیریت انواع مختلف ترافیک استفاده می‌کند:

```
درخواست‌های HTTP معمولی:
مرورگر → Nginx → Gunicorn → Django (WSGI)

WebSocket / بلادرنگ:
مرورگر → Nginx → Daphne → Django Channels (ASGI)
```

### نقشه سرویس‌ها (Docker Compose)

```
filmbaz_web            → Gunicorn (HTTP)
filmbaz_ws             → Daphne (WebSocket)
filmbaz_db             → PostgreSQL
filmbaz_redis          → Redis (کش + Channel Layer)
filmbaz_rabbitmq       → RabbitMQ (پیام‌رسان)
celery_support_worker  → Celery Worker
celery_beat            → Celery Beat (زمان‌بند)
nginx                  → پروکسی معکوس
```

### دو نقش Redis

Redis در این پروژه دو وظیفه مستقل دارد:

- **لایه کش** — کشینگ لیست فیلم‌ها و صفحات جزئیات با کلیدهای داینامیک SHA-256 برای کاهش فشار روی دیتابیس
- **Channel Layer** — مسیریابی پیام‌های WebSocket بین Django Channels consumers برای ارتباط بلادرنگ

## قابلیت‌ها

### 🎥 مدیریت فیلم و سریال

- کاتالوگ فیلم و سریال با متادیتای کامل (عنوان فارسی، عنوان اصلی، slug، توضیحات، امتیاز، تاریخ انتشار، کشور، مدت زمان، نوع محتوا، رده سنی)
- ریسایز و کراپ خودکار پوستر (`300×400`) و بک‌دراپ (`1600×900`) هنگام آپلود
- دسته‌بندی ژانر با روابط many-to-many
- بخش‌های صفحه اصلی برای جدیدترین و برترین فیلم‌ها
- مرتب‌سازی بر اساس امتیاز و تاریخ انتشار
- فیلتر بر اساس ژانر، سال انتشار و رده سنی

### 🔍 موتور جستجوی پیشرفته

- **جستجوی Trigram PostgreSQL** (افزونه `pg_trgm`) برای تطابق فازی و مبتنی بر شباهت
- تطابق تقریبی — جستجوی `Interstelar` نتیجه `Interstellar` را برمی‌گرداند
- رتبه‌بندی نتایج بر اساس میزان شباهت
- دقت بسیار بالاتر نسبت به جستجوی معمولی `icontains`

### 👥 سیستم بازیگران و عوامل تولید

- مدل‌های مجزا برای `Cast` (بازیگران) و `Crew` (کارگردان، تهیه‌کننده، نویسنده)
- سیستم نقش‌محور برای عوامل — قابل توسعه بدون تغییر ساختار دیتابیس
- روابط many-to-many از طریق مدل‌های واسط (`MovieCast`, `MovieCrew`)
- ساختار نرمال‌شده — هر بازیگر یا عامل تولید فقط یک‌بار ذخیره می‌شود
- ریسایز خودکار تصاویر پروفایل بازیگران و عوامل

### 👤 مدیریت کاربران

**احراز هویت**
- مدل کاربر اختصاصی (`FilmBazUser` با ارث‌بری از `AbstractBaseUser`)
- `FilmBazUserManager` اختصاصی برای کنترل کامل فرآیند ساخت کاربر
- ثبت‌نام، ورود، خروج
- بازیابی رمز عبور از طریق ایمیل با تولید توکن امن
- تغییر رمز عبور برای کاربران لاگین‌شده

**پروفایل**
- پروفایل قابل ویرایش (نام کاربری، ایمیل، شماره تلفن)
- آپلود عکس پروفایل با ریسایز و کراپ خودکار
- داشبورد شخصی

**اعتبارسنجی (سمت سرور)**
- نام کاربری: فقط حروف انگلیسی، اعداد و آندرلاین
- شماره تلفن: فرمت ایرانی ۱۱ رقمی شروع با `09`، بررسی یکتا بودن
- ایمیل: بررسی فرمت و یکتا بودن
- رمز عبور: بررسی تطابق با تکرار رمز

### 💾 لیست ذخیره‌شده‌ها (Watchlist)

- کاربران می‌توانند فیلم‌ها را ذخیره یا از لیست حذف کنند
- دسترسی به لیست ذخیره‌شده‌ها از داشبورد کاربری

### 💬 سیستم کامنت

- کامنت‌ها به فیلم‌ها و کاربران احراز هویت‌شده متصل هستند
- مرتب‌سازی بر اساس جدیدترین
- رندرینگ Ajax برای تجربه کاربری روان

### 🎫 سیستم تیکت

- کاربران می‌توانند بازخورد در سه دسته ثبت کنند: انتقاد، پیشنهاد، گزارش مشکل
- ارسال خودکار ایمیل تأییدیه پس از ثبت تیکت

### 📡 سیستم پشتیبانی آنلاین

برای جزئیات کامل بخش [سیستم پشتیبانی آنلاین](#سیستم-پشتیبانی-آنلاین) را ببینید.

## مقداردهی اولیه دیتابیس

فیلم‌باز یک pipeline واردسازی داده دارد که دیتابیس را از طریق Management Command های اختصاصی به صورت خودکار مقداردهی می‌کند. ترتیب اجرا به دلیل وابستگی‌های بین مدل‌ها اهمیت دارد:

```
1. load_genres       → ژانرها
2. load_casts        → پروفایل بازیگران
3. load_crews        → پروفایل عوامل تولید
4. load_movies       → فیلم‌ها و سریال‌ها
5. load_relations    → روابط فیلم ↔ بازیگر / عامل
6. load_pictures     → پوسترها و بک‌دراپ‌ها
```

این pipeline به صورت خودکار از طریق `entrypoint.sh` اجرا می‌شود، اما می‌توان آن را دستی هم اجرا کرد:

```bash
docker compose exec web python manage.py load_genres
docker compose exec web python manage.py load_casts
docker compose exec web python manage.py load_crews
docker compose exec web python manage.py load_movies
docker compose exec web python manage.py load_relations
docker compose exec web python manage.py load_pictures
```

## بهینه‌سازی‌های عملکرد

### کشینگ Redis

- صفحات لیست فیلم‌ها با کلیدهای داینامیک SHA-256 بر اساس پارامترهای Query (فیلترها، مرتب‌سازی، صفحه‌بندی) کش می‌شوند
- صفحات جزئیات فیلم و کامنت‌ها به صورت مستقل کش می‌شوند
- کش با به‌روزرسانی داده‌ها باطل می‌شود

### بهینه‌سازی Query

- از `select_related()` و `prefetch_related()` در سراسر پروژه استفاده شده تا مشکل N+1 Query در داده‌های رابطه‌ای (فیلم → ژانر، فیلم → بازیگر، فیلم → عامل) حل شود

### بهینه‌سازی تصاویر

- تمام تصاویر آپلودشده (پوستر، بک‌دراپ، عکس پروفایل، تصاویر بازیگران و عوامل) هنگام آپلود با `ResizedImageField` به صورت خودکار ریسایز و کراپ می‌شوند
- کاهش مصرف Storage و سرعت بارگذاری صفحات

### جستجوی Trigram PostgreSQL

- افزونه `pg_trgm` در سطح دیتابیس فعال شده
- جستجوی سریع و مبتنی بر شباهت بدون نیاز به موتور جستجوی خارجی

## سیستم پشتیبانی آنلاین

فیلم‌باز شامل یک چت پشتیبانی بلادرنگ Production-grade ساخته‌شده روی Django Channels است.

### نحوه کارکرد

```
کاربر
  │
  ▼
اتصال WebSocket
  │
  ▼
Daphne (سرور ASGI)
  │
  ▼
Django Channels Consumer
  │
  ▼
Redis Channel Layer
  │
  ▼
Consumer اپراتور پشتیبانی
```

### قابلیت‌ها

- **پیام‌رسانی بلادرنگ** از طریق `AsyncWebsocketConsumer` — بدون نیاز به رفرش صفحه
- **چت مبتنی بر نشست** — هر گفتگو در یک `SupportSession` با وضعیت `OPEN` / `CLOSED` ذخیره می‌شود
- **تاریخچه دائمی** — تمام پیام‌ها در `SupportMessage` با فرستنده، متن، زمان و وضعیت خواندن ذخیره می‌شوند
- **ردیابی خواندن** — فیلد `is_read` روی هر پیام برای نمایش وضعیت خوانده‌شده / نخوانده
- **اتصال‌های احراز هویت‌شده** — `AuthMiddlewareStack` تضمین می‌کند فقط کاربران لاگین‌شده می‌توانند کانال WebSocket باز کنند
- **گروه‌های کانال مستقل** — هر نشست یک گروه Redis مجزا دارد (`session_<id>`) تا پیام‌ها بین کاربران قاطی نشوند

### بستن خودکار نشست‌ها

نشست‌های پشتیبانی غیرفعال به صورت خودکار توسط یک وظیفه پس‌زمینه بسته می‌شوند:

```
Celery Beat (زمان‌بند)
  │
  ▼
RabbitMQ (پیام‌رسان)
  │
  ▼
Celery Worker
  │
  ▼
بستن خودکار نشست‌های قدیمی OPEN
```

### مقیاس‌پذیری

- Channel Layer مبتنی بر Redis از استقرار چند نمونه‌ای (Multi-Instance) پشتیبانی می‌کند
- Consumerهای async همزمانی بالا را بدون مسدود کردن Thread‌ها مدیریت می‌کنند
- کاملاً از ترافیک HTTP جدا از طریق سرور Daphne مستقل
